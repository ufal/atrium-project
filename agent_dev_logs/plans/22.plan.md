# Strategy — Issue #22: Document Understanding evaluation

> Issue [\#22](https://github.com/ufal/atrium-project/issues/22) · Labels: `question`, `beginning`, `a big one` · **Status: ✍️ In progress**

## Context
We need to measure our "document understanding" (transcription fidelity, layout, and semantic extraction). Initial discussions suggested OmniDocBench, but it is strictly for modern PDFs. Our domain is historical OCR and archaeological entities. The best domain-matched resource is **CHURRO-DS**. Rather than immediately building a complex pipeline that converts ALTO to Markdown to feed to an LLM (as suggested by Gemini DR), the strategy is to first run a head-to-head benchmark: **OOTB VLMs vs. our existing legacy pipeline**.

## Strategic actions 

### Milestone 1 — Curate & Annotate (The Dataset)
1. **Sample Selection**: Curate *N* representative archaeological pages.
2. **Stratification**: Use the existing `atrium-alto-postprocess` per-page quality score to stratify pages by difficulty/degradation tiers — implemented as `sample_stratify.py` (in `ufal/atrium-llm-enrich`): consumes `samples_page_stats.csv` or aggregates DOC_LINE_CATEG CSVs, buckets into tiers, deterministic 80/10/10 manifest.
3. **Annotation**: Generate gold transcriptions and optional entity tagging (aligning strictly to the existing CNEC/TEATER schema). Split into 80/10/10.

### Milestone 2 — The Head-to-Head Comparison
4. **Establish Baseline**: Run the curated dataset through the existing legacy pipeline (ABBYY ALTO + `atrium-nlp-enrich`).
5. **VLM Evaluation**: Run the dataset through localized open-weight VLMs (prioritizing CHURRO-3B). MiniMax-M3 can be run on the Grace Hopper node for a high-parameter ceiling check.
6. **Metric Reporting**: Calculate CER/WER, TEDS, and entity F1 per quality-tier (similar to the confusion matrix from page-classification) — scoring primitives implemented as `eval_metrics.py` (in `ufal/atrium-llm-enrich`; TEDS optional via `apted`+lxml).

### Milestone 3 — Tool Packaging
7. **`atrium-doc-understanding`**: Package the evaluation harness as a new tool repo, mirroring the existing CI/CD/paradata templates. 

## Tracking aids
- Use the methodologies developed in `atrium-page-classification` as the architectural template for setting up this evaluation repo.
- For heavy local inference (MiniMax-M3), coordinate with Viktor for the Grace Hopper node.

## Risks
- **Conflating Parsing with Understanding**: OCR layout (parsing) vs. NER/QA (understanding) require different metrics. Both must be measured, but start with parsing (CER/WER) as it dictates downstream success.
- **Markdown-Bridge Complexity**: Do not build the deterministic ALTO→MD→LLM→TEITOK pipeline until the baseline VLM comparison proves it is worth the engineering overhead.

## Out of scope
- **Model Fine-Tuning**: Focus purely on Out-Of-The-Box (OOTB) evaluations first. Fine-tuning a compact VLM is a stretch goal to chase SoTA later.
- **Dynamic Routing implementation**: Wait for metrics before building code to conditionally route high-confidence pages to deterministic extractors and low-confidence to VLMs.
