## Context

#13 is the **synthesis** issue. #10 built the ingestion (PDF/DOCX/ALTO/TEITOK → one page-sectioned,
cue-annotated Markdown; opt-in Tesseract `ces` OCR; a `page` field for citations) and #11 decided
the LLM diet is annotated Markdown (HTML = intermediate only; PAGE/ALTO/TEITOK = spatial source of
truth; TXT = fallback). #13 names the three formats that must coexist —

- **MD with HTML** — as LLM input,
- **TEITOK.XML** — as the visually-correct layout format,
- **JSON** — for search / text + metadata + any additional info storage —

and leaves two things to specify: **the level of detail in the HTML fragments of the Markdown** and
**a JSON schema template**. Today there is **no dedicated data-formats design doc** anywhere in the
ecosystem (`atrium-project/docs/` has `plan_repo_review.md`, `paradata-schema.md`,
`agent_skill_strategy.md`, but no format contract) and **no standalone JSON schema file** in any
repo — the enrichment JSON is only an in-code Pydantic model
([`llm_client_shared.build_document_schema`](../../llm_client_shared.py),
[`llm_run.build_schema`](../../llm_run.py)). This strategy fills both gaps as a proposal for the
assignees (david-spacil, motyc, rharasim, K4TEL) to work from.

Maintainer steer (2026-07-24): **document only**, delivered as an **issue comment** ("for now"),
**no pushes**; default HTML-fragment detail = **full `CUE_SCHEMA`**, lighter profiles selectable. The
posted comment is a lightly-trimmed "draft to work with" version of §A–§E below.

## §A — The three-plane model: one job per format

Each candidate format is authoritative for exactly one concern; no format serves two masters. The
planes are bound by paradata provenance (`run_id`) and by explicit cross-links.

| Plane                     | Format                                                                 | Role                                                                    | Authority                                                              | Produced by                                               |
|---------------------------|------------------------------------------------------------------------|-------------------------------------------------------------------------|------------------------------------------------------------------------|-----------------------------------------------------------|
| **Reading**               | **Annotated Markdown** (`# doc` + `## Page N` + `<!-- cues -->`)       | LLM input ("diet")                                                      | **Derived & disposable** — regenerable any time                        | `pdf_to_md`/`docx_to_md` · `xml_to_md.py --format layout` |
| **Layout / preservation** | **TEITOK.XML** (TEI + facsimile + bbox)                                | Human view, CQP/XPath search, correction, archival                      | **Spatial source of truth** — word/line coords + linguistic annotation | `teitok_alto.write_teitok_merged` (nlp-enrich)            |
| **Search / knowledge**    | **JSON** (§C template)                                                 | Text + metadata + entities + enrichment for indexing / catalogue export | **Canonical semantic record** — what search & ARIADNE/GoTriple consume | `run_document_level` (+ NER/quality merge)                |
| *(invariant)*             | **paradata JSON** ([`atrium_paradata`](../../atrium_paradata.py) v2.0) | Provenance / license / run binding                                      | **Immutable audit trail**                                              | every stage                                               |

**Directionality (never circular).** Markdown is always regenerable and never the thing we preserve;
**TEITOK is the persisted spatial artifact**, and the JSON aggregate (§C) is a *derived,
re-assemblable view* over the per-stage persistent outputs (§F.1) — both bound to the ALTO original
by `run_id`:

```
ABBYY ALTO ─┐
 OCR/VLM   ─┼─► TEITOK.XML ──(xml_to_md --format layout)──► annotated MD ──► LLM ──► JSON (§C)
PDF/DOCX  ──┘      ▲ spatial truth           reading plane                   │ knowledge plane
                   └─────────── deterministic re-projection (JSON → TEITOK <name>/notes) ◄─┘
 every stage ─► atrium_paradata  (run_id binds ALTO ↔ MD ↔ JSON ↔ TEITOK; license union)
```

This is the four-phase flow from [ufal/atrium-project#22](https://github.com/ufal/atrium-project/issues/22)
(Ingest → Markdown → JSON → TEITOK) stated as a contract, corrected to reality: the quality score is
a **post-extraction per-line** signal (not a pre-extraction router — `text_util_langID.py`), and the
JSON→TEITOK re-projection **already exists** (`atrium-nlp-enrich/api_util/teitok_alto.py`).

## §B — Spec 1: level of detail in the HTML fragments — full `CUE_SCHEMA` by default

The Markdown plane is a *derived, throwaway* view whose only job is to hand the model the richest
faithful picture of the page. So the default emits **every** layout cue we can recover, as inert
HTML comments a renderer hides and the LLM reads as positional hints. Detail is **profiled** so a
consumer can dial it down, but *full* is the default. The vocabulary is single-sourced in
[`api_util/layout_md.py`](../../api_util/layout_md.py) (`CUE_SCHEMA`: 15 HTML-comment cues in 5
groups + 8 inline GFM forms in `INLINE_CUES`).

| Group                      | Cue                                                                                      | Emitted form                                                                | Default (full)                                 |
|----------------------------|------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------|------------------------------------------------|
| Macro-structure            | `PAGE_BREAK`                                                                             | `<!-- PAGE_BREAK: pg_12 -->`                                                | ✅                                              |
|                            | `DOC_META`                                                                               | `<!-- DOC_META: size=612x792pt, orientation=portrait -->`                   | ✅                                              |
|                            | `LAYOUT_MARGIN`                                                                          | `<!-- LAYOUT_MARGIN: top=1in, … -->`                                        | ⚠️ reserved → to implement                     |
|                            | `LAYOUT_COLUMN`                                                                          | `<!-- LAYOUT_COLUMN: 1_of_2 -->`                                            | ⚠️ reserved → to implement                     |
| Spatial geometry           | `BBOX` (block-level)                                                                     | `<!-- BBOX: [x_min,y_min,x_max,y_max] -->`                                  | ✅                                              |
|                            | `INDENT`                                                                                 | `<!-- INDENT: 4px -->`                                                      | ⚠️ reserved → to implement                     |
| Typography                 | `FONT`                                                                                   | `<!-- FONT: size=14pt, weight=700, family="…" -->`                          | ✅ (size/family; **weight** reserved)           |
|                            | `STYLE`                                                                                  | `<!-- STYLE: color=#FF0000, highlight=yellow, line-spacing=1.5 -->`         | ✅ (color/highlight; **line-spacing** reserved) |
| Document regions           | `HEADER_START/END`, `FOOTER_START/END`                                                   | `<!-- HEADER_START --> … <!-- HEADER_END -->`                               | ✅                                              |
|                            | `WATERMARK`                                                                              | `<!-- WATERMARK: "CONFIDENTIAL DRAFT" -->`                                  | ⚠️ reserved → to implement                     |
| Pipeline signals           | `NEEDS_OCR`, `OCR`                                                                       | `<!-- NEEDS_OCR: pg_3 (…) -->` · `<!-- OCR: engine=tesseract, lang=ces -->` | ✅                                              |
| Inline (GFM, not comments) | bold/italic/strike/underline, `<div align>`, `![img]`+BBOX, `[^n]` footnotes, GFM tables | native GFM                                                                  | ✅ all                                          |

**Two invariants that keep "full" faithful rather than bloated:**
1. **Aggregate at the highest logical level** — `BBOX`/`FONT` per *block/paragraph*, never per word.
   Word/line coordinates live in the TEITOK plane, not in Markdown. This bounds token cost and avoids
   the coordinate-as-tokens sequence blow-up.
2. **Cues are positional hints, never content** — the document system prompt (`_DOC_SYSTEM_HEADER` in
   [`llm_client_shared.py`](../../llm_client_shared.py)) already tells the model to read cues for
   position but "never extract or quote them as content." Full detail must not change what counts as
   text.

**Selectable down-profiles** (a `--detail` knob over the same `CUE_SCHEMA`, kept available, not the
default):
- **`standard`** — drop typography (`FONT`/`STYLE`/`LAYOUT_MARGIN`/`INDENT`/`WATERMARK`); keep
  structure + `BBOX` + `DOC_META` + header/footer + columns.
- **`minimal`** — structure only: page breaks, headings, GFM tables, lists, footnotes, image
  placeholders, `NEEDS_OCR`/`OCR`. Guarantees page-citation + reading order at near-zero overhead.

**Empirical gate (doesn't change the default; quantifies its cost).** Run full/standard/minimal
through the #22 bake-off (`sample_stratify.py` → converters → `bench_compare.py` with a
**tokens-per-page** column, scored by CER/WER + entity-F1, TEDS for tables). If full-detail's tokens
hurt accuracy/budget on a tier, the profile knob pins a lighter default *per tier* without changing
the format.

*So "full" is honestly full,* the ⚠️ reserved cues (`weight`, `INDENT`, `LAYOUT_COLUMN`,
`LAYOUT_MARGIN`, `STYLE` line-spacing, `WATERMARK`) still need emitters in
`pdf_to_md`/`docx_to_md`/`xml_to_md` — the vocabulary exists in `layout_md.py`, only the extraction
wiring is missing (deferred, §E).

## §C — Spec 2: the JSON schema template (the search/knowledge plane)

**Principle: harmonise, don't invent.** The template is the union of what already exists —
`build_document_schema` fields, the on-disk `*_enriched.json` records, the paradata provenance block,
the CNEC/OntoNotes/TEATER taxonomies, the quality bands, and the TEITOK/ALTO spatial refs — as one
canonical, versioned, **FAIR** document record (stable `doc_id`, controlled vocab, ontology
cross-refs, cross-walk to sibling formats).

Top-level object **`AtriumDocument`** (illustrative; *optional* layers may be externalised by
reference):

```json
{
  "schema_version": "1.0",
  "doc_id": "CTX000000001",

  "source": {
    "filename": "CTX000000001.alto.xml",
    "media_type": "application/alto+xml",
    "sha256": "…",
    "page_count": 12,
    "language": ["cs"],
    "origin": "ABBYY-ALTO"
  },

  "derived_from": {                     // ONLY persistent step outputs — never transient artifacts
    "classification": "CATEG/CTX000000001.categories.json",
    "alto_categ":     "DOC_LINE_CATEG/CTX000000001.csv",
    "translated_xml": "TRANSLATED/CTX000000001.alto.xml",
    "teitok":         "TEITOK/CTX000000001.teitok.xml",
    "enriched":       "KW_PER_DOC_LLM/CTX000000001_enriched.json",
    "paradata":       "paradata/260724-101112_pipeline-run.json"
  },
  "regenerable": {                      // disposable derivations — a recipe, not a stored path
    "markdown": { "from": "TEITOK/CTX000000001.teitok.xml",
                  "converter": "xml_to_md@0.3.0", "detail": "full" }
  },

  "provenance": {

                        "is_share_alike": true, "components": [ "…" ] }

  "assembled": {                        // this aggregate is a VIEW, re-mergeable per block (§F.1)
    "mode": "view",
    "source_run_ids": { "pages": "260724-071648", "lines": "260724-071648",
                        "entities": "260724-090012", "enrichment": "260724-101112" }

      "needs_ocr": false, "ocr": null,

    "reading_order": "ltr-columns"
  },

  "lines": [
    { "page": "1", "line": 14, "categ": "Text",

      "type_onto": "FAC", "type_cnec": "gc", "type_teitok": "LOC",
      "page": "1", "line": 14, "char_span": [21, 37], "bbox": [340, 400, 470, 432],
      "teitok_ref": "CTX000000001.name5", "translation_en": "gothic church",

      { "locator": "gotického kostela ze 14. století",
        "page": "1",

        "extracted_keywords_en": ["foundations", "gothic church"],
        "teater_category": "kostel",
        "confidence_score": 0.95,
        "citation": "[Source: CTX000000001, Page 1]" }

    "summary": "…optional document summary…",
    "topics": ["…"],
    "page_categories": { "1": "Text", "2": "Plate" }
```

**Field lineage (why each block exists / where it comes from today):**
- `schema_version` — mirrors paradata's `SCHEMA_VERSION` policy
  ([`atrium-project/docs/paradata-schema.md`](https://github.com/ufal/atrium-project/blob/test/docs/paradata-schema.md)):
  additive fields need no bump; renames/semantics bump the version + ship a `_migrate_X_to_Y()`.
- `source` — the **original input** (always persistent). `origin` records treatment (`ABBYY-ALTO` |
  `digital-born-pdf` | `docx` | `ocr:tesseract-ces` | `ocr:pero` | `vlm:glm-4v`).
- `derived_from` / `regenerable` — the **FAIR cross-walk**, restricted to what actually persists (§F.2):
  `derived_from` points only at **persistent step outputs** (classification / alto categ / translated
  XML / TEITOK / enriched.json / paradata); `regenerable` records *disposable* derivations (Markdown,
  and by the same rule any page image) as a **recipe** (source + converter + `--detail`), never a
  stored path. **No page-image/thumbnail references** — those are transient (§F.2).
- `provenance` — **references** the paradata record (no duplication); carries the effective license so
  the JSON is self-describing for catalogue export (`para_licenses.py` union).
- `assembled` — marks the record as a **derived view** and maps each block to the stage `run_id` that
  produced it, so a single stage can be updated and only its block re-merged (§F.1). `mode` is `view`
  (assembled on demand) or `cache` (materialised, staleness-checked against per-stage paradata).
- `pages[]` — `quality_score`/`quality_band` from `atrium-alto-postprocess/text_util_langID.py::compute_quality_score`
  (Trash `<0.55` / Noisy `[0.55,0.80)` / Clear `≥0.80`); `canvas` from `DOC_META`; `ocr`/`needs_ocr`
  from the OCR cues; `teitok_surface` links into the TEITOK facsimile.
- `content` — full text or a pointer to the annotated MD; keeps the JSON a usable search document
  without inlining megabytes.
- `lines[]` *(optional)* — mirrors nlp-enrich's CSV columns (`page/line/categ/quality_score/text`) and
  TEITOK `<s>`/`<lb>`; `teitok_ref` is the join back to spatial truth.
- `entities[]` — normalised NER carrying **all three tagsets** (`type_onto` OntoNotes = default
  configured NameTag model, `type_cnec` fine legacy, `type_teitok` coarse PER/ORG/LOC/MISC), the LLM
  `translation_en`, and a nullable `pid` block for **FAIR ontology cross-refs**
  (Wikidata/GeoNames/AAT/AMCR) — the ARIADNE/GoTriple hook.
- `enrichment` — a superset of `build_document_schema.DocumentEnrichment` (`locator`, `page`,
  `extracted_keywords_cs/en`, `teater_category`, `confidence_score`) + the explicit `citation` anchor
  (#11 §3.3), optional `summary`/`topics`, and `page_categories` from
  [`atrium-page-classification`](https://github.com/ufal/atrium-page-classification).

**Versioning & validation.** Ship alongside the template a JSON-Schema (draft 2020-12) file
`atrium_document.schema.json` for CI validation, generated from a pydantic `AtriumDocument` that
extends `build_document_schema` (one generator, no drift). *Where the file lives* —
single-sourced/hub-canonical like `atrium_paradata.py` (para-drift-enforced) vs owned by llm-enrich —
is **deferred** (§E); specified here for now.

## §D — What #13 exposes (flagged, not undertaken — this is a design issue)

1. **The Markdown plane exists in only one repo.** `layout_md`/`xml_to_md`/`doc_to_visual_md` live in
   `atrium-llm-enrich`; **`atrium-nlp-enrich` still feeds its LLM plain CSV lines / `.teitok.xml`**
   (`llm_run.py`), and the LLM engine (`llm_utils`/`vocab_manager`/`llm_run`) is duplicated &
   diverging across the two, **outside `para-drift`** (`plan_repo_review.md` §5.M). Unifying nlp-enrich
   onto annotated-MD input is the biggest downstream consequence of adopting this strategy.
2. **TEITOK correctness items** (for "TEITOK.XML for *correct* layout"): reconcile the current code's
   `xmlnsoff=`/`lang=` vs the committed samples' `xmlns=`/`xml:lang=`; make `<name>` closing tags
   consistent (or always run `bbox_scale.fix_name_close_tags`); replace the hard-coded
   `<div type="MarginTextZone-P">` with the real ALTO block role. (`teitok_alto.py`.)
3. **Provenance already the right shape** — no new format needed; `provenance` references the existing
   paradata `run_id`, and `merge_run_paradata.intermediate_formats` already enumerates the formats a
   run touched.

## §F — Refinements from review round 1 (motyc / David Novák, 2026-07-24)

Two concerns on the posted draft, both about the **production persistence model** — which is exactly
the not-yet-built orchestration (hub #18: per-stage containers over a shared `atrium_data` volume,
`plan_repo_review.md §8`). Folded in as follows.

**§F.1 — The aggregate must not couple the pipeline into a single full run.**
> *"it means the pipeline will only work as a full run, and updating individual parts will also
> require regenerating this JSON in full … consider this when finalising the … production pipeline."*

The `AtriumDocument` is **not** a monolithic store; it is a **derived, re-assemblable view** over
the per-stage persistent outputs, which remain the source of truth and stay independently
produced/updated (matching the per-stage-container model of #18):
- Each stage keeps writing its own persistent output (classification result · alto categ/quality CSV ·
  translated XML · **TEITOK** · `enriched.json`) + its own paradata — nothing forces a full run.
- The aggregate is built by a **deterministic merger** — the *same* pattern
  `atrium_paradata.merge_run_paradata` already uses to fold per-stage paradata into one summary
  ("Effective license/intermediate_formats reflect EXECUTED stages only"). The `assembled.source_run_ids`
  map tags each block with the stage `run_id` that produced it.
- **Incremental update:** re-run one stage → its output + `run_id` change → re-merge **only that block**
  (a near-instant read+copy, no LLM/NER/OCR recompute). "Update individual parts" therefore needs the
  one stage plus a trivial re-merge — never a full pipeline run.
- **Two materialisations**, pick per production need: `mode:"view"` (the aggregator service, §F.3,
  assembles it on request — nothing extra persisted, zero staleness) or `mode:"cache"` (materialised
  `…document.json`, staleness detected by comparing block `run_id`s to the latest per-stage paradata).
- Skipped stages degrade gracefully (block absent/null), mirroring the paradata convention.

**§F.2 — Reference only persistent artifacts; no page-image/thumbnail refs.**
> *"we cannot and should not keep any reference to 'images' (page thumbnails) … temporary artefacts …
> not stored. The only references … are the original files or the persistent outputs of previous steps."*

- **Dropped** `representations.images` and `pages[].image`. References are restricted to the two
  persistent classes David names: the **original input** (`source`) and **persistent step outputs**
  (`derived_from`).
- **Markdown**, being *derived & disposable* (§A), is likewise **not** a stored path — it moves to
  `regenerable` as a reproducible recipe (source TEITOK/ALTO + converter version + detail profile).
  The image rule and the Markdown rule are the *same* principle: reference persistent sources +
  recipes, never transient derivatives.
- **Visual overlay still works without stored images:** bboxes stay in ALTO/TEITOK coordinate space;
  the presentation layer (TEITOK web platform / thumbnails created late in the pipeline) re-renders a
  page image **on demand** and overlays via TEITOK's `<surface>` facsimile ids. `pages[].teitok_surface`
  is a *logical* id, not an image path.

**§F.3 — The aggregate is its own tool + service, fed the inputs (motyc, follow-up 2026-07-24).**
> *"The tools will be separate and they won't see all the actual outputs of individual steps. If we
> want the JSON as an aggregate, it needs its tooling and running service to produce it from given
> inputs."*

Correct — and it matches the settled ecosystem stance (hub #18: **no central runtime orchestrator**;
each tool is a self-contained GHCR container **reached via its own API**, `service/api.py`,
§4.2 of `agent_skill_strategy.md`). No existing stage sees the others' outputs, so the aggregate
cannot be a byproduct of one. It is a **new, standalone component** — an *aggregator*
(`POST /aggregate`, following the same standardized service + zero-dep-client + Agent-Skill contract
as the five tools) that is **given** the persistent outputs as inputs (originals + classification
JSON + alto categ CSV + translated XML + TEITOK + `enriched.json` + each stage's paradata) and emits
the `AtriumDocument`. It is **stateless and deterministic** — same inputs → same JSON — so it does not
depend on shared-volume visibility; the caller (compose smoke, a batch job, or the presentation layer)
hands it the artifacts. This is the sixth-service option in the ecosystem research; it also subsumes
the "manifest" idea already present as `nlp-enrich/api_util/build_manifest_row.py`
(`doc_id⇥page_count⇥textfile`), which the aggregator generalises into the full record.

## Verification
- **Self-consistency:** every format the issue names is assigned exactly one plane (§A); every JSON
  field names its source in current code (§C lineage); the full-detail default is stated with its
  safety invariants and its empirical gate (§B).
- **Round-trip sanity (described, runnable later):** `xml_to_md.py --format layout` on a sample ALTO
  shows the full-detail cues while `--format markdown` stays byte-identical; `run_document_level`
  emits an `enrichment` block that maps 1:1 onto `AtriumDocument.enrichment`; a hand-built
  `AtriumDocument` validates against the drafted JSON-Schema.
- **Delivery:** strategy posted as a comment on this issue (attribution footer appended). **No
  commits, no pushes** to any branch.
