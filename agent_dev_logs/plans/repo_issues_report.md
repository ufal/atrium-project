# **Strategic Infrastructure, Governance, and Development Roadmap for the ATRIUM Project Repositories**

The ATRIUM project operates at the forefront of computational linguistics, optical character recognition (OCR) refinement, and semantic enrichment, targeting the complex domain of historical and archaeological archives1. The architectural landscape is distributed across a specialized suite of repositories, each managing discrete segments of the data pipeline: atrium-project serves as the administrative and governance hub, atrium-alto-postprocess handles the triage and structural refinement of raw XML files, atrium-nlp-enrich executes complex large language model (LLM) pipelines for named entity recognition (NER) and semantic mapping, and atrium-translator facilitates the in-place multi-modal translation of the enriched artifacts1.  
Given the reliance on autonomous algorithmic execution and the transient nature of active agent development logs across test branches—which are frequently ephemeral or inaccessible post-execution4—the establishment of immutable project management artifacts is a critical engineering requirement. The systematic generation of dedicated plans and digests directories ensures that the overarching tracking, legal compliance, and algorithmic transparency of the repository ecosystem remain structurally intact. This report provides an exhaustive, detailed overview of the newly created issue-specific plans and digests, mapping the highly complex operational logic and cross-repository continuous integration milestones.

## **Global Operations, Governance, and Academic Structuring**

The atrium-project repository functions as the foundational administrative architecture backing the entire development lifecycle2. Because this environment handles the heavy overarching tracking for deliverables and cross-functional tasking, the prior omission of a dedicated plans directory constituted a significant governance gap. Resolving this required the formulation of highly detailed structural documents to map administrative, legal, and academic objectives with the same rigor applied to the software engineering processes8.

### **Legal Provenance and License Mapping: Issue 6**

The integration of disparate open-source software tools, machine learning frameworks, and highly protected archaeological datasets introduces severe licensing complexities. The project operates within the broader context of the LINDAT/CLARIAH-CZ repository infrastructure, which enforces strict distribution license agreements and complex intellectual property frameworks9. Consequently, addressing Issue 6 required the immediate creation of 6.plan.md to map out the legal constraints, operating alongside the existing digest to document ongoing compliance tracking8.  
The 6.plan.md artifact establishes a comprehensive review protocol for all deployed models and resulting data artifacts. The LINDAT digital repository mandates that any deposited content must be accompanied by sufficient metadata describing its provenance, and this metadata must be unconditionally distributed into the public domain under the Creative Commons Zero (CC0) license9. However, the underlying data and the software used to process it operate under a deeply fractured landscape of legal constraints. The plan dictates a systematic audit to resolve incompatibilities, specifically focusing on the deployment of tools like UDPipe, MorphoDiTa, and the translation frameworks2.  
The legal matrix formulated within 6.plan.md categorizes assets based on their distribution requirements. Software components released under the GNU General Public License (GPL-2.0 or GPL-3.0) permit distribution and modification but mandate the disclosure of the source code for any derivative application12. In contrast, the NameTag 3 Named Entity Recognition framework, a critical component of the downstream enrichment pipeline, operates under a dual-license architecture: the software itself is governed by the Mozilla Public License 2.0, while the associated linguistic models are strictly restricted to non-commercial use under the CC BY-NC-SA license13.

| Asset Category                  | Primary License Framework      | Compliance Requirements and Constraints                                                                                |
|:--------------------------------|:-------------------------------|:-----------------------------------------------------------------------------------------------------------------------|
| **Repository Metadata**         | Public Domain Dedication (CC0) | Complete waiver of copyright; freely accessible metadata for provenance and format description without restriction.    |
| **NameTag 3 Software**          | Mozilla Public License 2.0     | Permits distribution and modification; requires original copyright attribution.                                        |

| **Core Distributed Codebases**  | GPL-2.0 / GPL-3.0              | Requires tracking of source code modifications; mandates full source code disclosure for any distributed applications. |

To navigate these constraints, 6.plan.md includes rigorous instructions for the automated generation of paradata logs. As the Python scripts process documents, they must generate unified provenance metadata saved in structured JSON formats14. These paradata logs capture comprehensive execution details, including the specific tool names, repository URLs, Python environment versions, and unique run identifiers, ensuring that any researcher downloading the derived archival datasets is provided with a consolidated, legally sound provenance record that satisfies LINDAT's requirements for security measures protecting proprietary copyright15.

## **OCR Triage, ALTO XML Processing, and Quality Categorization**

The atrium-alto-postprocess repository acts as the fundamental data ingestion, triage, and sanitization layer for the entire architectural ecosystem. It is explicitly responsible for transforming raw, document-specific ALTO XML files into structured page-level statistics, reconstructing reading orders, and rigorously filtering low-quality optical character recognition noise1. Because downstream natural language processing models are highly susceptible to hallucinations when fed corrupted or fused tokens, the mathematical rigor of the post-processing filters is paramount.

### **Algorithmic Definition of Text Categories: Issue 2**

Issue 2 represents a critical, fundamental update to the text category definitions, specifically implementing strict quality-score thresholds and multi-modal computational signals17. Because this update dictates the foundational routing logic for all subsequent data, it absolutely required plans/2.plan.md to map the algorithmic design prior to implementation, along with digests/2.digest.md to document the quantitative outcome of the logic update.  
The 2.plan.md document meticulously defines the logic for categorizing every extracted line into one of five immutable classes: Clear, Noisy, Trash, Non-text, or Empty1. The computational architecture leverages a highly efficient CPU/GPU split to process these categorizations at scale. A single, dedicated GPU worker holds the Qwen2.5-0.5B instance, processing text batches to calculate neural perplexity scores while preventing Virtual RAM Out-Of-Memory errors1. Simultaneously, up to 32 concurrent CPU workers execute memory-bound file reading operations, FastText language identification, and deterministic structural detection logic1.  
The plan dictates a strict sequential evaluation protocol. Initially, minor, universally applicable OCR repairs are executed. Following this, the CPU workers evaluate the line against a hierarchy of deterministic bypass exceptions and standard rules. A line is immediately flagged as Empty if it contains exclusively whitespace characters1. The Non-text categorization is significantly more complex, designed to filter out archival administrative noise that holds no linguistic value for the archaeological text mining pipelines. Lines are flagged as Non-text if they consist entirely of digits, arithmetic separators, and punctuation, or if they match Roman numeral patterns1. Furthermore, the plan defines rigorous regular expressions to catch standalone alphanumeric archive codes—such as "CTX200205348"—and stamp-like ratio patterns such as "1998/01"1. Finally, extreme brevity or a lack of alphabetic density triggers a Non-text classification; lines with fewer than four total characters, or where alphabetic characters comprise less than 30% of the total string, are categorically rejected1.  
Lines surviving this deterministic pre-filter are then subjected to the generative quality assessment. The composite quality score is an aggregated float value mapped between 0 and 1, combining structural detectors, neural perplexity, and character-level metrics1. The FastText semantic confidence plays a pivotal role here. The pipeline configuration defines EXPECTED\_LANGS (such as Czech, German, and English) and TRUSTED\_FOREIGN\_LANGS1. If the FastText confidence score falls below the LANG\_SCORE\_ROUGH threshold of 0.45, the input is considered too degraded to trust, heavily penalizing the composite score1. Conversely, lines subjected to collection-default force-remapping are assigned a stabilized minimum floor defined by LANG\_SCORE\_CLEAR at 0.751.  
Ultimately, lines achieving a high composite score are categorized as Clear and are forwarded to the NLP enrichment workflows1. Lines with moderate scores—exhibiting isolated symbol issues, mid-word uppercase anomalies, or elevated perplexity indicating fused tokens—are marked Noisy, signaling that corrective algorithms are necessary before semantic processing1. Lines falling below the baseline threshold are permanently classified as Trash and routed for re-processing by alternative OCR engines1. Following the execution of this logic, the 2.digest.md document captures the empirical output shifts, validating the efficacy of the post-processing smoothing mechanisms, particularly the inverted-scan sweep that successfully identifies fundamentally unreadable pages based on aggregate FastText failures1.

### **Calibration Against Beta Strings: Issue 3**

To ensure the theoretical categorization parameters defined in Issue 2 operate accurately against highly anomalous historical data, Issue 3 necessitates rigorous calibration against a verified dataset of beta strings1. The generation of plans/3.plan.md was required to outline the exact calibration methodology and define the evaluation metrics before any dynamic testing began.  
The 3.plan.md document details the precise extraction frameworks deployed. It utilizes the alto-tools framework alongside the LayoutReader architecture to parse the bounding boxes of XML elements, specifically targeting \<TextLine\> elements that contain String elements with the CONTENT attribute1. The calibration strategy focuses heavily on reconstructing the reading order to accommodate complex, multi-column historical print layouts. Crucially, the methodology explicitly defines the handling of hyphenated words split across line breaks, mandating the algorithmic reconstruction of the full morphological form to prevent artificial spikes in neural perplexity scoring1. The plan establishes fixed quantitative thresholds for precision and recall that the composite scoring system must achieve on the beta validation set, tuning the harmonic weights of the 10 combined signals to ensure robust performance prior to merging the branch into the production environment1.

## **Semantic Enrichment, Ontology Mapping, and LLM Execution**

The atrium-nlp-enrich repository represents the most intellectually demanding and computationally intense segment of the architecture. It is tasked with executing advanced cross-lingual semantic understanding, morphological parsing, and dynamic named entity extraction from historically complex texts2. Establishing the plans directory within this repository was an absolute necessity to thoroughly document the heavy architectural decisions involving agentic LLM pipelines, ontological frameworks, and strict application programming interface schemas.

### **TEATER Keyword Extraction and Local LLM Pipeline: Issue 6**

The extraction of highly specific, domain-restricted archaeological vocabulary from historical Czech texts into a rigid, structured JSON template represents an intersection of advanced machine learning challenges, including instruction adherence, morphological robustness, and the mitigation of generative hallucination14. Consequently, plans/6.plan.md was authored to rigorously define the constrained decoding strategy and the dynamic prompt formulation mechanisms required to execute the TEATER topics pipeline.  
The plan initiates by defining the ingestion protocols for discrete text lines previously marked as Clear by the ALTO pipeline. Because the underlying historical text has already undergone preliminary layout-aware extraction via architectures like LayoutLMv3 and generative baseline correction via GLM-4v-9b, the pipeline must ensure the prompt context is optimally configured14. Supplying the LLM with the entirety of the multi-megabyte TEATER thesaurus within the system prompt would immediately exhaust the context window, dilute the model's attention mechanism, and trigger catastrophic hallucination14. To circumvent this, the plan mandates a Dynamic Vocabulary Injection strategy. A semantic similarity search, utilizing a highly optimized embedding model such as all-MiniLM-L6-v2, actively harvests the document chunk against the nested dictionary structure (teater\_nested\_vocab.json), which is categorized by broad archaeological domains including "Site Types", "Find Types", "Chronology", and "Methods"14. Only the top-N most relevant hierarchical dictionary trees are serialized and dynamically injected into the system prompt.  
The execution environment relies on local inference engines running via Ollama. The 6.plan.md document extensively logs the benchmarking of various base models to determine the optimal deployment target. The evaluation revealed highly divergent behaviors regarding JSON schema adherence and vocabulary hallucination14.

| Model Architecture      | Inference Efficiency | Behavioral Characteristics and Schema Mechanics                                                                                                                                   |
|:------------------------|:---------------------|:----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Ministral 3 14B**     | Baseline (1.0x)      | Flawless Pydantic parsing via native agentic optimization; however, exhibited severe extraction failures, defaulting to empty results or nonsensical geographical distance tags.  |
| **Qwen 2.5 14B**        | High (0.7x)          | Excellent multilingual translation and high extraction density; requires specific "Non-Thinking Mode" to bypass internal logit conflicts and prevent formatting degradation.      |
| **Bielik 11B v3.0**     | High (0.8x)          | Exceptional factual grounding but susceptible to repetitive loops on ambiguous lines; requires custom term-priority prompt sorting to prevent defaulting to "Country" categories. |
| **Aya Expanse 8b**      | Low                  | Prone to severe hallucination; frequently misclassified standard administrative addresses as obscure sovereign states (e.g., East Timor, Liechtenstein).                          |
| **DeepSeek-R1-Distill** | Unpredictable        | Critical pipeline failure; internal reasoning tags collided violently with the logit masking enforcers, breaking the JSON schema entirely.                                        |

To enforce output strictness, a Pydantic model is defined within the Python runtime, serializing a rigid blueprint containing the original\_text, extracted\_keywords, teater\_broad\_type, teater\_narrow\_type, and a confidence\_score14. The inference engine utilizes LogitMatch masking, guaranteeing that the generated output populates the DocumentEnrichment template flawlessly, drawing exclusively from the permitted vocabulary subsets14. To counteract the behavioral flaws observed in models like Bielik 11B—which exhibited the "Country Default" issue where ambiguous text caused the model to randomly select nations from the "Other" category—the plan implements a rigorous \_term\_priority logic. This algorithm assigns numerical weights to thematic domains (e.g., Chronology \= 9, Site Types \= 8), ensuring that the most highly relevant terms are heavily prioritized in the prompt attention mechanism, stabilizing the deterministic output14.  
Furthermore, 6.plan.md introduces contextual window optimizations, shifting away from isolated 2-line processing to a "Section Detection" paradigm, informing the LLM of the overarching document section (e.g., "Description of Finds") to drastically improve thematic mapping consistency across disjointed text blocks14.

### **Domain-Specific NER and NameTag 3 Integration: Issue 7**

Standard off-the-shelf named entity recognition models are incapable of parsing the deeply nested, highly specific terminologies found in archaeological surveys. Issue 7 demands the integration of the NameTag 3 framework to execute domain-specific NER based on the ArchaeoBERT model guidelines19. The plans/7.plan.md document clearly outlines the complex ontological guidelines and maps out the computational training phases.  
NameTag 3 is an advanced, open-source tool designed for both flat and nested NER. Unlike flat tagging which merely isolates sequential entities, nested tagging requires identifying multiple subtypes within major entity boundaries. To achieve this, NameTag 3 replaces standard classification heads with a sophisticated sequence-to-sequence (seq2seq) decoding head, achieving state-of-the-art performance across multilingual datasets13.  
The 7.plan.md document transcribes the ArchaeoBERT ontological philosophies into algorithmic extraction rules. The definitions are highly restrictive and strictly enforced19.  
The ARTEFACT category is defined broadly to include any object found in the ground, ranging from pollen and skeletal remains to massive structural elements like wooden canoes and building beams19. However, the rules explicitly forbid the tagging of overly generalized categories such as "archeobotanical material" or "find complexes". The handling of descriptive adjectives is highly nuanced; adjectives that describe the physical state of the artefact must be included within the entity boundary (e.g., "burnt wattle and daub", "rough-walled sherd"), whereas adjectives that merely describe the material composition must be excluded19.  
The PERIOD definitions require temporal precision while stripping linguistic noise. Dates featuring specific days and months must be reduced to just the year entity (e.g., "12 december 2012" is tagged solely as "2012")19. For multi-year spans, the surrounding prepositional context is maintained ("between approx. 1700 and 1850"), but broad introductory words like "before," "after," or "circa" are strictly truncated from the annotation19. The guidelines further dictate that general temporal contexts like "the Carolingian Empire" are valid period entities, whereas purely relative designations like "Later periods" are ignored entirely19.  
Spatial mapping under the LOCATION category requires the removal of administrative prefixes; the phrase "Municipality Zutphen" is annotated simply as "Zutphen"19. The rules explicitly ignore geographical directions unless they are formalized parts of the location name, and completely reject the tagging of coordinates, rivers, seas, or broad designations like "the Dutch coastal area"19.  
The MATERIAL entities are strictly limited to describing the composition of an identified artefact, explicitly forbidding the annotation of the adjectival form (e.g., "molten bronze" is truncated to "bronze")19. Finally, the SPECIES guidelines navigate the complexity of human remains; while "human bones" requires the word "bones" to be tagged as an artefact, the generalized concept of human influence is excluded from biological tagging, and Latin taxonomies followed by vernacular names require separate, disjointed annotations19.  
The plan also details the operational deployment of NameTag 3\. To maintain data privacy for highly sensitive archaeological location data, the pipeline bypasses the public REST Web Service, instead utilizing local Python clients (nametag3\_client.py) to execute the models directly within isolated environments, managing arguments for model directories, specific tagsets, and required licensing acknowledgments13.

### **API Formulations and Ingestion Standardization: Issues 8 and 10**

The utility of the advanced NLP enrichment architecture is entirely constrained by its ability to ingest diverse historical file structures and dynamically serve the resulting data. Issue 8 addresses the requirement for a programmatic interface for TEITOK XML streams20. The plans/8.plan.md document constructs the OpenAPI schema, detailing the payload validation mechanics required to securely accept XML documents. It maps the staging design, detailing how the incoming TEITOK formatted data is buffered, routed through the layout reconstruction matrix, processed by the NameTag 3 seq2seq heads, and serialized back into enriched JSON payloads. The companion digests/8.digest.md summarizes the implementation outcome, specifically documenting the performance overhead introduced by parsing image file dependencies intrinsically linked within the TEITOK architecture20.  
Concurrently, Issue 10 manages the chaos of raw text ingestion20. Because archival data arrives in highly fragmented, legacy formats, handling this diversity justifies the creation of both plans/10.plan.md and digests/10.digest.md. The plan outlines the integration of flexiconv support, acting as an abstraction layer to standardize any text-containing file format into the internal unified denominator string before semantic enrichment is attempted20. The resulting digest documents the engineering challenges overcome during implementation, particularly the resolution of severe character encoding anomalies, non-standard line breaks, and hidden control characters that frequently disrupt the deterministic structural pre-filters in the upstream pipelines.

## **Architectural Modularization of Machine Translation**

The concluding segment of the ATRIUM processing workflow focuses on accessibility. To ensure that deeply domain-specific text extracted from Czech, German, or Dutch archives can be utilized by the broader international archaeological community, the text must be accurately translated into English while preserving the precise XML structural tagging3. The atrium-translator repository is dedicated to this in-place XML manipulation.

### **Pluggable Translation Backends: Issue 4**

The original architecture relied on a static, monolithic wrapper for translation tasks. Transitioning this architecture to a dynamic, multi-backend support system supporting both LLM-driven endpoints and localized CTranslate2 engines represents a task of high architectural complexity8. This transition completely justified establishing the plans directory, resulting in the creation of a dedicated plans/4.plan.md to map out the modularization strategy, alongside digests/4.digest.md to track the execution.  
The 4.plan.md document details the comprehensive decoupling of the translation logic from the XML parsing framework. It defines an abstract interface class capable of routing designated text blocks based on continuous evaluation metrics. For environments with high computational availability, the backend can route to external LLM APIs, capable of maintaining high contextual fidelity over long document spans. Conversely, for environments requiring extreme throughput or operating under strict data-privacy mandates, the interface routes to the internalized CTranslate2 inference engine.  
The plan heavily dictates the parameter mapping protocols. It ensures that critical execution variables—such as dynamic batch sizing, beam search width for decoding optimization, and the enforcement of source-language token prefixes—are seamlessly standardized and correctly formatted regardless of which backend is active. Crucially, the plan governs the in-place replacement mechanics. The translated English strings must be re-injected into the document tree without violating the highly validated, schema-compliant structures of the ALTO or AMCR XML files3. Any deviation or formatting error during the string injection phase would render the file unreadable by subsequent archival storage systems.  
The associated 4.digest.md captures the empirical results of this modularization. It records the performance benchmarks, explicitly contrasting the latency profiles of CTranslate2 when executing large-batch parallel processing on CPU clusters versus dedicated GPU acceleration1. Furthermore, it validates the generation of paradata, ensuring that the precise identity, versioning, and execution parameters of the specific translation backend utilized are permanently appended to the document's metadata JSON, ensuring absolute scientific traceability throughout the pipeline's lifecycle14.

## **Strategic Synthesis**

The orchestrated deployment of the plans and digests directories across the ATRIUM project repositories marks a critical maturation in the project's software engineering and administrative lifecycle. By cementing atrium-project as the central locus for legal compliance mapping and academic objective tracking, the overarching governance of the digital ecosystem is secured. The mathematically rigorous categorization definitions established in atrium-alto-postprocess guarantee that the neural pipelines are insulated from corrupted data. Within atrium-nlp-enrich, the exhaustive ontological formulation for NameTag 3 and the highly constrained schema enforcement for local LLMs dictate a standard of semantic precision that mitigates generative hallucination. Finally, the structural decoupling achieved in atrium-translator ensures future-proof, hardware-agnostic multi-lingual dissemination. Together, these meticulously documented architectural blueprints ensure the ATRIUM infrastructure operates with unparalleled transparency, traceability, and computational efficiency.
