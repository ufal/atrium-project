# Strategy — Issue #26: Running models larger than GPU memory using CPU offload

> Issue [#26](https://github.com/ufal/atrium-project/issues/26) · Labels: `development` · Milestone: Q4 [WP3] · **Status: 🔄 In progress (was: ⛔ blocked)**

## Context
Run models whose parameter footprint exceeds available VRAM by spilling weights into CPU RAM —
the "unified memory mechanism" the issue asks to explore and apply in practice. The project-state
digest flags #26 as **blocked pending an architectural decision**.

That decision is, in effect, already made — and the mechanism is **already implemented** in
`atrium-nlp-enrich`. The issue thread concludes that **vLLM's `cpu_offload_gb` (UVA — Unified
Virtual Addressing: zero-copy CPU-RAM offload where the GPU stays the sole compute engine)** is
the right fit for ATRIUM's profile (one node, single/few GPUs, 20+ CPU cores already contended by
the FastText / regex / queue workers). It is preferred over (a) Ollama/llama.cpp `num_gpu`
layer-split, which offloads *compute* and steals CPU cycles from those workers; (b) raw CUDA UVM
(`GGML_CUDA_ENABLE_UNIFIED_MEMORY`), immature outside Grace-Hopper NVLink hardware; and (c) HF
Accelerate `device_map="auto"`, relevant only to the small Transformers-native models.

The mechanism is wired end-to-end already:
- **`llm_config.txt`** — documents a `CPU_OFFLOAD_GB` knob with concrete node recipes
  (e.g. `qwen3-235b-a22b-fp8`, `TENSOR_PARALLEL_SIZE=4`, `CPU_OFFLOAD_GB=70` on 4×L40 48 GB).
- **`llm_utils.py`** — `_GLOBAL_INFERENCE_FALLBACKS["cpu_offload_gb"]=0`,
  `_PARAM_KEYS["CPU_OFFLOAD_GB"]→"cpu_offload_gb"`, resolved + int-coerced in
  `get_inference_defaults()`, forwarded by `load_vllm_engine()` →
  `engine_kwargs["cpu_offload_gb"]` (with startup logging) when `> 0`.
- **`llm_run.py`** — reads the resolved value, prints it in the inference-parameter summary, and
  passes it to `load_vllm_engine()`.

So #26 is **not greenfield**: it is "validate the existing capability on real hardware, expose it,
document it, then formally unblock." This plan enumerates the remaining work.

## Strategic actions

### A — Validate in practice (closes "applied in practice")
Run one concrete over-VRAM scenario from `llm_config.txt`'s recipes — e.g.
`qwen3-235b-a22b-fp8` with `TENSOR_PARALLEL_SIZE=4` + `CPU_OFFLOAD_GB=70` on dll-4gpu3 (503 GB
RAM), or a dense 70 B with offload on a single 48 GB card. Run `llm_run.py` and capture: the
startup inference-parameter summary, `log_gpu_memory()` before/after load, the paradata JSON, and
the `avg in/out tok/s` line. This run *is* the deliverable that moves #26 off "blocked."

### B — Hardware preflight (establish the ceiling)
On each candidate node record `nvidia-smi --query-gpu=name,memory.total --format=csv`,
`nvidia-smi topo -m`, and PCIe gen/width. Offloaded weights cross PCIe (~tens of GB/s) vs HBM
(~2–3 TB/s) on every forward pass, so the interconnect sets the practical penalty (offload docs
estimate **3–6×**). Decide the per-node `CPU_OFFLOAD_GB` budget from VRAM headroom + measured
penalty.

### C — Auto-offload heuristic (future code work — scoped, not done here)
Estimate weight footprint vs. detected VRAM and auto-set `cpu_offload_gb` (with a warning) instead
of relying on manual config; optionally add an offload value to specific large-model
`inference_defaults`. Reuse the `total_memory` read already in `log_gpu_info()`.

### D — Documentation (future)
Add a README **"Running models larger than VRAM (CPU offload)"** subsection and an offload
note/column in the hardware tables; cross-link `llm_config.txt`. Keep the framing honest: this is
*virtual* extra VRAM with a per-pass transfer cost, not free VRAM — quantization/AWQ stays the
primary strategy, offload is the exception.

### E — transformers-backend decision
The default BnB backend has no CPU-spillover path. Either (i) wire `device_map="auto"` +
`max_memory` spillover, or (ii) **document "over-VRAM ⇒ use `BACKEND=vllm`"** as the supported
answer. **Recommend (ii)** — less code, and vLLM is already the large-model path.

### F — Coordinate with #27 (one decision matrix)
Offload (#26) and tensor-parallel / expert-parallel sharding (#27, H100 / MiniMax-M3) are
complementary: use TP first to consume all local VRAM, then offload to top up the remainder. The
235 B / 444 B cases need both. Document them together rather than as competing options.
