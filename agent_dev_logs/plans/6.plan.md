# Strategy — Issue #6: Review & summarise tool/model licenses

> Issue [#6](https://github.com/ufal/atrium-project/issues/6) · Labels: `documentation`, `enhancement` · **Status: 🔍 Tocheck**

## Context
The license inventory is **done** and has been propagated into two live surfaces: per-run
paradata (shared `atrium_paradata.py` / `para_licenses.py`, all four repos) and the SSHOMP
**tool** records (license tables appended to each description). The remaining work is
documentation-completeness and keeping the inventory honest as backends change — not new
research. The one genuinely open item is attaching the license list to the SSHOMP
**workflow** records (distinct from the tool records, which are done).

This plan exists because #6 is the canonical legal-constraint map for the whole ecosystem:
the **CC-BY-NC-SA** chain it documents (NameTag3 + CUBBITT + layoutreader) is the exact
constraint that translator [#4](https://github.com/ufal/atrium-translator/issues/4) is now
trying to lift, so #6 must stay in lock-step with every backend swap.

## Strategic actions

### A — Close the open TODO: licenses on the SSHOMP *workflow* records
1. Reuse the exact tool-record tables (already written) for the four workflow records.
2. If the workflow listing keeps 500-ing (seen in #4), escalate to the SSHOMP admins rather
   than blocking — record the attempted URL and the error in the issue so it is auditable.
3. Mark each workflow record's license block with the **source repo + version** so a reader
   can tell which release the license set describes (mirrors the F2 `date-released` hygiene
   from `docs/plan_repo_review.md`).

### B — Make the inventory drift-proof (tie to the hub review)
The `para_config.txt` component list is the machine-readable twin of this digest's table.
- Add a check (or a periodic manual reconciliation) that every component named in any repo's
  `para_config.txt` appears in #6's inventory with a matching license string — so a new
  backend (e.g. `eurollm`, `madlad400`) can never reach a release without its license being
  reviewed here first.
- Where translator #4 introduced the deliberately-unrecognised `llm_api = "LLM provider ToS"`
  string, flag it in #6 as **needs a licensing-owner decision before non-experimental use**
  (it currently resolves to maximally-restrictive on purpose).

### C — Record the "permissive output" recipe centrally
#6 should hold the one authoritative statement of what it takes to ship a **permissive**
output, because it spans repos:
- translation model permissive (EuroLLM / MADLAD, Apache-2.0), **and**
- FastText langid avoided (explicit `--source_lang`, since FastText is CC BY-NC), **and**
- permissive/empty glossary (AMCR/TEATER vocab are CC BY-NC).
This is already written in translator #4's `translation-backends.md` design — link it from #6
so the legal map and the implementation recipe never diverge.

## Verification / done-criteria
- All four SSHOMP **workflow** records carry the license table (or the blocker is documented).
- Every `para_config.txt` component across the four repos is represented in #6's table.
- The CC-BY-NC-SA chain and the permissive recipe are stated once, in #6, and linked from
  translator #4 and hub #9/#16.
