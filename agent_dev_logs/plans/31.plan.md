# Strategy — Issue #31: AGENT SKILL — based on API service to each repo

> Issue [#31](https://github.com/ufal/atrium-project/issues/31) · Labels: `enhancement`, `development` · Milestone: Q3 [WP8/WP7] · **Status: 🟢 Strategy authored — implementation via sub-issues**

## Context
Wrap each service repo's API in an **Agent Skill** (`SKILL.md` per the
[open standard](https://agentskills.io) — the cross-vendor format adopted by Claude Code, Codex,
Copilot, Antigravity, Gemini CLI, …), on a dedicated **`agent-skill` branch** per repo.
`atrium-page-classification` already carries the exemplar branch: root `SKILL.md` +
zero-dependency stdlib client (`scripts/atrium_classify.py`) + idempotent `scripts/server.sh` +
flattened/trimmed source tree + `small_data_samples/`. The full normative strategy — service
meta-contract, branch anatomy, client contract, SKILL.md template, per-repo work plans — is
**[`docs/agent-skill-strategy.md`](../../docs/agent-skill-strategy.md)**; this plan is the
issue-level index of it.

Decisions locked at planning time:
1. **llm-enrich gets a minimal FastAPI `service/` first** (mirrors nlp-enrich) — all five repos
   share one server–client standard.
2. **Meta-contract standardization only**: uniform `GET /info` + `GET /health`, multipart
   conventions, error-code table (413/415/422/429/500/503), env vars (`ALLOWED_ORIGINS`,
   canonical `MAX_UPLOAD_MB`, client `ATRIUM_<XX>_URL`), version from `para_config.txt`.
   Domain endpoints (`/predict_*`, `/translate`, `/process`, `/enrich`, new
   `/extract_keywords`) are **not** renamed.
3. **No committed OpenAPI snapshots** — agents fetch the live `/openapi.json`; drift is caught
   by per-repo contract tests + a future `skill-validate.reusable.yml` in the hub.

## Strategic actions

### A — Publish the standard (this session)
`docs/agent-skill-strategy.md` in the hub: §4 service meta-contract · §5 branch anatomy ·
§6 zero-dep client contract (3× retry on 502/503/504, exit codes 0/1/2/3) · §7 SKILL.md
template + anti-pattern checklist (the 4 exemplar defects, generalized into rules) ·
§9 frontend/README requirements (API footer: curl + `/docs` + `/openapi.json` links) ·
§10 per-repo checklists (sub-issue-ready) · §12 drift control.

### B — Sub-issues (next)
One per repo under #31, titled `agent-skill: <repo>`, body lifted from §10 + pinned-commit link
to the doc. Implementation on each repo's `agent-skill` branch (create from default-branch
head; page-classification updates the existing branch).

### C — Rollout order
1. page-classification hardening (small; makes the exemplar trustworthy) →
2. nlp-enrich (validates templates → promote Appendices A–D to `docs/templates/skill/`) →
3. alto-postprocess ∥ translator pre-work →
4. translator skill branch; llm-enrich (service first, skill after) →
5. `skill-validate.reusable.yml` in the hub once ≥2 skill branches exist (frontmatter check;
   referenced-paths-exist check; client `--help` in bare `python:3.11-slim`; optional
   endpoint diff vs `app.openapi()`).

### D — Naming (locked)
Branch `agent-skill` everywhere · skill `name:` = repo name · clients `atrium_classify.py`
(exists) / `atrium_translate.py` / `atrium_postprocess.py` / `atrium_enrich.py` /
`atrium_keywords.py` · env vars `ATRIUM_{PC,TR,AP,NE,LE}_URL`.

## Verification path

locally started server; `/info`+`/health` conform to §4.1; end-state acceptance = a clean-room
