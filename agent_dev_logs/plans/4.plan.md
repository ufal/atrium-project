## Strategic actions

### A — Confirm the tool records (the `Tocheck` part)
1. Spot-check each default tool version actually **displays** the appended license table
   (admin-updated versions only show on the default version — easy to miss).
2. Verify the **ATRIUM catalogue** keyword returns all four tools in one query, so the
   catalogue is discoverable as a set, not four disconnected records.

### B — Resolve or park the workflow-record gap
The workflow listing 500-errored; two clean outcomes, pick one and record it:
- **Resolve:** report the 500 to SSHOMP admins with the exact query URL
  (`…/search?…&categories=workflow&f.keyword=ATRIUM+catalogue&…`); once listable, attach the
  same license tables (ties into #6 action A).
- **Park:** if the registry genuinely can't host the workflow records cleanly, document the
  decision ("keep tool records canonical; workflows tracked in-repo via the compose pipeline
  from #18") so the gap is an explicit choice, not a silent omission.

### C — Keep records in sync with releases
Tool versions on SSHOMP are admin-updated and only the default version is shown. Add a
lightweight reminder to the release checklist: on a version bump that changes the tool's
license set, re-submit the SSHOMP description table (the machine-readable twin is
`para_config.txt`, kept honest by #6).

## Verification / done-criteria
- Four tool records resolve, display license tables on their default versions, and surface
  under the ATRIUM-catalogue keyword.
- The workflow-record outcome is recorded in #4 (resolved **or** deliberately parked).
