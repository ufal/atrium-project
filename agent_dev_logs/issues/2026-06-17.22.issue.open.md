Gemini DR report conclusions:

The strategy is predicated on the foundational principle that no single data representation can possibly satisfy the requirements of archival preservation, linguistic query systems, and LLM inference engines simultaneously. The architecture must seamlessly manage transformations between these states while rigorously adhering to the strict data governance and structured logging standards mandated by the ATRIUM infrastructure.


### Phase 1: Ingestion, Triage, and Algorithmic Quality Routing
The pipeline begins with the ingestion of the ABBYY Finereader ALTO XML files alongside their corresponding high-resolution archival images. Immediate application of the atrium-alto-postprocess module is necessary. This script operates as the primary gatekeeper, evaluating the baseline optical character recognition quality of the legacy ALTO files.
By mathematically calculating a composite quality score derived from perplexity metrics and structural coherence algorithms, the system performs an automated triage. Documents that successfully meet a high-confidence threshold are routed to a deterministic, highly efficient textual extraction path. Documents that fall below the threshold—indicating severe physical degradation, highly complex tables, or archaic typography that thoroughly confused the legacy ABBYY engine—are dynamically routed to a heavy Multimodal Visual path. This dynamic routing algorithm prevents the catastrophic financial and computational cost of passing all millions of historical pages through a massive vision model, reserving the expensive image-based inference strictly for the specific documents that require it.


### Phase 2: Format Transformation and Token Abstraction
Depending on the specific routing decision, the document is transformed into the LLM-optimized input format.


The high-confidence ALTO XML data is processed by a specialized conversion utility, leveraging the architectural concepts proven by tools like Docling or Marker, to extract the raw text strings and algorithmically rebuild the reading order based strictly on the HPOS and VPOS coordinates. The output of this conversion is strict, semantically clean, highly token-efficient Markdown. Tables defined by the ALTO spatial blocks are carefully reconstructed into standard Markdown tables, allowing for row and column intersection calculations. Hierarchical text block sizes and font metadata are utilized to infer the correct placement of Markdown headers.
Conversely, for low-confidence documents, the raw document image is passed directly to an optimized Multimodal Large Language Model. The prompt precisely instructs the model to perform Document Layout Analysis and text extraction simultaneously, outputting the result directly as structured Markdown, thereby bypassing the broken textual representation entirely.
Both computational paths ultimately converge, resulting in a perfectly clean, token-efficient Markdown representation of the historical document, fully optimized for natural language processing and entirely devoid of XML syntactic noise.


### Phase 3: The Document Understanding Inference Execution
With the complex historical document successfully abstracted into lightweight Markdown, the core Document Understanding tasks can be executed with maximum efficiency. This involves submitting the Markdown payload to the designated local or remote language model alongside highly specific system prompts.


For the ATRIUM infrastructure, these automated tasks will encompass complex semantic enrichment, identifying abstract concepts, historical entities, or obscure geopolitical references that require deep contextual reasoning that extends far beyond the limited capabilities of standard deterministic Named Entity Recognition scripts. The models may also be utilized for cross-lingual translation, executing processes akin to the modular Python wrapper designed for the LINDAT Translation API, which translates identified entities into English while safely preserving structural metadata. Finally, the model performs broad summarization and key information extraction, generating vital metadata parameters suitable for indexing in massive catalogues such as the ARIADNE Knowledge Base or GoTriple.


Crucially, the interaction with the language model must remain entirely stateless and focused purely on inference. The model processes the clean Markdown, computes the complex historical relationships, and returns structured JSON or strictly formatted output detailing its precise findings, structural classifications, or extracted historical entities.


### Phase 4: Deterministic Re-integration into Archival and Linguistic Systems
The final architectural phase directly addresses the ATRIUM project's mandate for rich, interoperable, and persistent data structures. The data generated by the language model during inference cannot remain in isolated, flat JSON files. It must be mathematically mapped back onto the original spatial and linguistic architecture to be useful to humanities scholars.


A highly deterministic, rule-based computational script must take the model's output and project it back into the persistent storage formats. For the linguistics and digital humanities researchers, the baseline text from the ALTO files is computationally merged with the semantic classifications and standard entity tagging to generate a fully compliant, heavily annotated teitok.xml file. This specific transformation allows the historical documents to be permanently hosted on the TEITOK web platform, directly enabling researchers to visualize the historical text alongside its image, search the massive corpus using the Corpus Query Processor, and manually correct algorithmic hallucinations in a highly user-friendly interface.


Simultaneously, the pipeline executes rigorous Provenance Logging. The ATRIUM infrastructure demands a heavy emphasis on data traceability and operational transparency. Every single step of the pipeline—ranging from the initial ALTO quality score, to the specific Markdown conversion logic applied, the exact LLM model and version utilized for inference, and the temperature parameters of the prompt—must be systematically logged into a structured JSON file within a dedicated paradata directory. A unique run_id cryptographically binds the original legacy ALTO file to its newly generated TEITOK derivative, ensuring that future researchers and auditors understand exactly how the machine-readable text was extracted, transformed, and semantically enriched.
This sophisticated, multi-stage architectural blueprint ensures that the computational power of generative artificial intelligence is harnessed with maximum token efficiency, while entirely preserving the linguistic depth, structural fidelity, and rigorous provenance tracking demanded by the global digital humanities community.

### Summary: Document Understanding Strategy for Historical ALTO XML Corpora

**Benchmarks:** The OmniDocBench framework, referencing both arXiv:2412.07626 and arXiv:2603.04205, is an effective candidate for this evaluation. Tool evaluations utilizing this framework demonstrate that compact, domain-specialized Vision-Language Models outperform massive generalized models when processing noisy historical documents.

**Model Evaluation Strategy:** It is strongly recommended to evaluate open-source, locally executed models rather than enterprise APIs to maintain data sovereignty and enable precise paradata logging. Furthermore, while Multimodal Large Language Models (MLLMs) can bypass OCR to achieve higher accuracy (89.1% versus 76.8%), running millions of pages through MLLMs is financially and computationally unfeasible.

**Data Formatting & Annotation Scheme:**
* Feeding raw ALTO XML directly to LLMs should be avoided, as the token-heavy structural metadata causes "attention dilution".
* Instead, data must be transformed into Markdown, which is the optimal format for balancing token efficiency with the retention of spatial and hierarchical structures.
* The final annotated output must be deterministically mapped back into TEITOK XML, which is the necessary format for human visualization, rich linguistic annotation, and Corpus Query Processor (CQP) interfacing.


**Pipeline Methodology:**
* The pipeline must begin with algorithmic triage using the `atrium-alto-postprocess` module to calculate a baseline OCR quality score for the legacy ALTO files.
* High-confidence documents will be routed to an efficient text-extraction path, reserving the heavy MLLM processing strictly as a fallback for low-confidence, complex, or highly degraded pages.
* Language models will then execute stateless inference on the Markdown payloads to perform key information extraction and semantic enrichment, outputting structured JSON.
* Finally, a rule-based parser will re-integrate this JSON data back onto the original spatial architecture while executing rigorous provenance logging to bind the derivative files to the original ALTO data.

## Bottom line up front
The Gemini "Deep Research" comment reads well but **conflates two different problems and contains several fabrications and mis-attributions**. The two biggest issues:

1. **"Document understanding" ≠ what OmniDocBench measures.** OmniDocBench scores *parsing/transcription fidelity* (OCR, layout, tables, reading order). Semantic *understanding* (NER/KIE/QA/summarization) is a separate axis — and it's largely what `atrium-nlp-enrich` **already does**. These need different datasets, metrics, and models. The plan must separate them.
2. **OmniDocBench does not contain historical/archaeological material.** It's modern PDFs. Adopting it alone would not measure your actual domain. The historical-document resource that *does* match — and that the report missed — is **CHURRO / CHURRO-DS**.

## Part 1 — What the report got wrong (and right)

| Claim | Verdict | Reality |
|---|---|---|
| `atrium-alto-postprocess` is a "gatekeeper" doing OCR-quality **triage/routing** | **FALSE** | It's a *linear* pipeline (split → stats → extract → quality-classify → aggregate), `run_pipeline.py`. No conditional routing exists anywhere. |
| Quality score = "perplexity + **structural coherence algorithms**" | **MISLEADING** | A composite score *does* exist (`text_util_langID.py:552-636`) but from **10** signals — valid-word-ratio 25%, garbage-density 20%, **perplexity (Qwen2.5-0.5B) 15%**, symbol 13%, word-weirdness 13%, etc. "Structural coherence" is invented. It's **per-line, computed after extraction** — not a pre-extraction routing signal. |
| We must "build" a fallback multimodal path | **PARTLY ALREADY EXISTS** | A **GLM-4v** image-extraction method already ships (one of three: alto-tools / LayoutReader / GLM-4v), CLI-selectable. Only the *per-document decision logic* is missing. |
| `nlp-enrich` has a "**LINDAT Translation API** wrapper" | **FALSE** | No translation wrapper there. It calls LINDAT **UDPipe** (morphology) + **NameTag 3** (NER, CNEC 2.0). English translation is done by the **LLM itself** (`llm_run.py`). A separate `atrium-translator` repo exists, but it's not this pipeline. |
| "standard **deterministic** NER scripts" | **MISLEADING** | NameTag 3 / UDPipe are **neural** models. |
| TEITOK XML + paradata/run_id + license union | **CORRECT ✓** | `api_util/teitok_alto.py` (TEI + facsimile + bbox) and `atrium_paradata.py` are real and accurately described. The report's "Phase 4 re-integration" is **already built**. |
| MLLMs hit "**89.1% vs 76.8%**" by skipping OCR | **FABRICATED** | Untraceable to any source. Real literature: OCR-free VLMs *can* match/beat pipelines on some tasks, but there's no universal figure — it's document/task/language-dependent. |
| "Compact specialized VLMs beat massive general models on noisy historical docs" | **TRUE but MIS-SOURCED** | Correct in spirit — but the evidence is **CHURRO** (3B beats Gemini 2.5 Pro on historical OCR at 15.5× lower cost) and **SmolDocling** (256M ≈ 7B-class), *not* OmniDocBench/Real5 as the report implies. |
| OmniDocBench is the benchmark for "our" docs | **SCOPE MIS-STATED** | OmniDocBench (CVPR'25) = **modern** PDF parsing (9 modern doc types). `arXiv:2603.04205` (Real5-OmniDocBench — I verified it's **real**, 2026-03-04) only physically *degrades* those modern images (scan/warp/illumination/skew). Neither covers historical/archaeo content. |

One nuance worth keeping: the report's instinct toward **local open-weight models for data sovereignty** is sound, and its **TEITOK + paradata + provenance** emphasis matches what you've already built. The architecture isn't wrong — it's just over-claimed and pointed at the wrong benchmark.

## Part 3 — Alternative approaches & their complexity

| Decision | Options | Recommendation | Complexity |
|---|---|---|---|
| **What to measure** | A) parsing/transcription (CER/WER, TEDS, mAP) · B) semantic understanding (entity F1, EM, ROUGE) | **Both, staged — start with A.** Everything downstream depends on transcription, and your data is OCR'd historical print. Stop conflating them. | A: med · B: med-high |
| **Dataset** | A) reuse public as-is · B) build in-domain gold set · C) hybrid | **C** — report public numbers for comparability + a small in-domain gold set for validity. Reuse classification curation/split tooling. | med-high |
| **Models / mode** | A) end-to-end VLM (image→text) · B) your existing ABBYY+ALTO+nlp-enrich pipeline · C) Markdown-bridge (ALTO→MD→LLM, the report's plan) | **Run A vs B head-to-head first** — the high-value question is "does an OOTB VLM beat the legacy pipeline on *our* pages?" C is a later optimization. | A: med · B: low (exists) · C: high |
| **Quality-based triage** (report's centerpiece) | build doc-level routing on existing per-page scores | **Defer.** Feasible, but it's a *production cost-optimization*, not the benchmark question the issue asks. | med |
| **OOTB vs finetune** | start OOTB; finetune later | **OOTB now; finetune a compact VLM (CHURRO-style) as the "improve SoTA" stretch** — exactly your framing. | OOTB: low · FT: high |
| **Local vs API** | report says local-only | **Local for production (sovereignty is valid); but include 1–2 frontier APIs in the benchmark** as a topline ceiling — cheap and informative. | low |

The reason C (the Markdown-bridge) is "high complexity" is that it requires deterministic ALTO→Markdown reconstruction *and* the JSON→TEITOK re-projection — a lot of brittle mapping code — for a benefit that's unproven until you've run the A-vs-B comparison.

## Part 4 — Recommended methodology (mirroring your image-classification work)

Reuse the `atrium-page-classification` recipe (curate → annotate → 80/10/10 split → multi-model compare → per-category/confusion breakdown → optional ensemble → package with paradata):

1. **Task + annotation scheme** — primary: full-page transcription (reading-order text) + layout/region labels; optional secondary: KIE/NER aligned to your existing **CNEC/TEATER** schema so results plug straight into `nlp-enrich`.
2. **Dataset** — curate *N* representative archaeo pages, **stratified by your existing quality score** (difficulty/degradation tiers), annotate gold transcriptions (+ optional entities), 80/10/10 split.
3. **Metrics** — CER/WER + normalized edit distance (OmniDocBench-compatible), TEDS for tables, a reading-order score; entity F1 for understanding. Report **per page-type / quality-tier** — the analogue of your confusion-matrix breakdown.
4. **Model comparison** — the OOTB line-up (CHURRO, Qwen2.5-VL, InternVL, olmOCR, GOT-OCR2, SmolDocling, dots.ocr, PaddleOCR-VL) **vs the ABBYY/ALTO pipeline baseline**; optional ensemble (your classification repo already has the tooling).
5. **Engineering** — package as a new `atrium-doc-understanding` tool reusing the shared Docker/CI/paradata/license template + CLI + FastAPI, like the other repos; log every run via paradata.
6. **Stretch** — finetune a compact VLM on the in-domain gold set to chase SoTA (this is where you'd beat off-the-shelf, per CHURRO).

**Concrete first step:** stand up the **OOTB-VLM vs legacy-pipeline** comparison on a small, quality-score-stratified sample of archaeo pages, scored with CER/WER + TEDS. That single experiment answers the issue's real question and tells you whether the heavier Markdown-bridge/triage machinery is even worth building.
