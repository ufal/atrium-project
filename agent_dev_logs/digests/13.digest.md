## 🎯 Goal
Turn the issue's three candidate formats — **MD+HTML** (LLM input), **TEITOK.XML** (visually-correct
layout), **JSON** (search / metadata / additional info) — into one coherent, FAIR data-format
architecture, and nail down (a) how detailed the HTML fragments in the Markdown should be, and (b) a
JSON schema template. Neither a data-formats design doc nor a standalone JSON schema exists in the
ecosystem yet — this fills both.

## 🧭 The answer in one picture
**Three planes, one job each** (bound by paradata `run_id` + explicit cross-links):
- **Reading — annotated Markdown** (`# doc` + `## Page N` + `<!-- cues -->`): the LLM diet, **derived
  & disposable**, regenerable any time from the layout plane. Produced by `pdf_to_md`/`docx_to_md` and
  `xml_to_md.py --format layout`.
- **Layout / preservation — TEITOK.XML** (TEI + facsimile + bbox): the **spatial source of truth**;
  human view, CQP/XPath search, correction, archival. `atrium-nlp-enrich/api_util/teitok_alto.py`.
- **Search / knowledge — JSON `AtriumDocument`**: the **canonical semantic record** for indexing /
  ARIADNE/GoTriple export; harmonises the existing enrichment + paradata + entity schemas.
- **Directionality is never circular:** Markdown regenerates from TEITOK; **TEITOK is the persisted
  spatial artifact and the JSON is a derived, re-assemblable view** (§F) over the persistent step
  outputs, cross-linked; every stage logs `atrium_paradata`. This is #22's
  Ingest → MD → JSON → TEITOK flow stated as a contract (and corrected: the quality score is a
  *post-extraction* per-line signal, not a router; JSON→TEITOK re-projection *already exists*).

## 🔎 Spec 1 — HTML-fragment detail: **full `CUE_SCHEMA` by default**
Markdown is a throwaway view, so emit **every** recoverable layout cue as inert HTML comments
(`layout_md.CUE_SCHEMA`: 15 comment cues + 8 inline GFM forms). Kept faithful (not bloated) by two
invariants: **aggregate at block level** (word/line coords stay in TEITOK) and **cues are positional
hints, never content**. Selectable down-profiles `standard` (drop typography) and `minimal`
(structure only) via a future `--detail` knob; the empirical token-cost check runs through #22.
Several cues are **defined-but-reserved** (`weight`, `INDENT`, `LAYOUT_COLUMN`, `LAYOUT_MARGIN`,
`STYLE` line-spacing, `WATERMARK`) — vocabulary exists, emitters pending.

## 🔎 Spec 2 — JSON schema template: **`AtriumDocument`** (harmonise, don't invent)
One canonical, versioned, FAIR document record unioning today's pieces, and — per review §F — a
**derived, re-assemblable view** (not a monolith) that references **only persistent artifacts**:
`schema_version` · `source` (original input) · `derived_from` (persistent step outputs only) +
`regenerable` (Markdown/images as a **recipe**, never a stored path) · `provenance` (paradata `run_id`
+ license union) · `assembled` (per-block `source_run_ids` → re-merge one stage without a full run) ·
`pages[]` (quality band, canvas, OCR, `teitok_surface` — **no image path**) · `content` · `lines[]`
(nlp-enrich CSV + TEITOK `<s>`/`<lb>` join) · `entities[]` (**all three tagsets** OntoNotes/CNEC/coarse
+ `translation_en` + nullable `pid` ontology refs) · `enrichment` (superset of
`build_document_schema.DocumentEnrichment` + explicit `citation`). Ship a draft-2020-12
`atrium_document.schema.json` from a pydantic model extending `build_document_schema`; distribution
(hub-canonical vs llm-enrich-owned) and the view-assembler are deferred to the #18 orchestration.

## ⚠️ What #13 exposes (flagged, not fixed)
- **MD plane is single-repo** — nlp-enrich still feeds plain CSV/`.teitok.xml`; the LLM engine is
  duplicated & diverging across nlp-/llm-enrich, outside `para-drift`. Unifying it is the biggest
  downstream consequence.
- **TEITOK fidelity bugs** — `xmlnsoff=` vs `xmlns=`; `<name>` close-tag inconsistency
  (`fix_name_close_tags` exists); hard-coded `<div type="MarginTextZone-P">`.
- **Provenance already right** — reuse `atrium_paradata`; no new format.
