## 🎯 Goal
Run models whose parameter size exceeds available VRAM by using CPU RAM — explore and apply a
"unified memory mechanism" in practice.

## 🧵 Decision trail
- **2026-06-20, K4TEL** — opened: figure out how to run over-VRAM models using CPU memory; the
  unified-memory mechanism is to be explored and applied.
- **2026-06-21, K4TEL (research comment)** — recommended **vLLM `cpu_offload_gb` (UVA)** as the
  best production-supported fit, drawing the key distinction:
  - **Compute offload** (Ollama/llama.cpp `num_gpu` split) — runs spilled layers *on CPU cores*,
    competing with the FastText/regex/32-worker queue already on the node.
  - **Memory-only offload** (vLLM UVA / raw CUDA UVM) — GPU stays the **only** compute engine;
    weights sit in pinned CPU RAM, read via a unified address space. A 24 GB GPU set to offload
    10 GB behaves like a 34 GB GPU.
  - For ATRIUM's contended-CPU profile, **memory-only is the better fit** — grows effective VRAM
    from system RAM without stealing cycles from the CPU workers.
  - Nuance flagged: vLLM maintainers **declined** general CUDA Unified Memory oversubscription
    (#10267, not-planned). What ships is the narrower, opt-in, per-GPU `cpu_offload_gb` budget —
    *not* transparent automatic paging.

## ✅ Outcomes (recommendation)
- **Mechanism:** `vllm serve … --cpu-offload-gb N --gpu-memory-utilization 0.90` (effective VRAM =
  physical + N GB).
- Keep **AWQ / quantization as the primary strategy**; offload is the exception, not the steady
  state. Every offloaded weight pays a per-forward-pass PCIe transfer cost — "virtual," not free,
  extra VRAM.
- Comparison: vLLM UVA = best match · Ollama split = competes for cores · `GGML_CUDA_ENABLE_
  UNIFIED_MEMORY` = immature off Grace-Hopper · HF Accelerate `device_map="auto"` = only the small
  Transformers models.

## ⏳ Open
- Confirm fast CPU↔GPU interconnect (PCIe Gen4/5 x16) on the target node before committing.
- Adopting vLLM framed as "a real infra decision" — **note:** vLLM is *already* a backend in
  `atrium-nlp-enrich`, and `CPU_OFFLOAD_GB` is already wired end-to-end, so this is narrower than
  the comment implies. See [`../plans/26.plan.md`](../plans/26.plan.md).
